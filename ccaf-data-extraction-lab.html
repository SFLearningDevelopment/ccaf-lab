Turn documents into
schema-valid JSON.

A single hands-on capstone: a structured data extraction system on the Claude API. It reads unstructured documents — invoices, contracts, resumes — and emits JSON that validates against a schema every time, handles the messy edge cases without fabricating, and hands clean records to whatever runs downstream. You design the schema, constrain the output, and build the validation loop that makes it reliable.

Before you start

You should be comfortable calling an API and reading JSON Schema — this lab assumes the developer experience the exam itself assumes. It does not stop to explain what a schema is or how to make an HTTP request.

  • Python 3.10+ or Node 18+ with the Anthropic SDK, and an API key exported as ANTHROPIC_API_KEY. Examples lead in Python.
  • A schema validator for post-checks — jsonschema (Python) or ajv (Node). You’ll validate even constrained output, for reasons Stage 5 makes clear.
  • A handful of real, messy documents of one type (say, five invoices) — including at least one with a missing field or an ambiguous value, so the edge-case stages have something to bite on.
Verify before you build Structured outputs moved to general availability recently, and the exact surface is still settling. The field name (output_config.format), whether a beta header is needed, the supported JSON Schema subset, and strict-tool-use flags all carry real shelf-life. Wherever you see a Verify note, check it against the current docs at platform.claude.com/docs. The shape of each pattern is stable; exact identifiers drift.
Orientation

What you’re building, before you build it

By the end of this lab you’ll have one working pipeline: a program that takes an unstructured document, extracts the fields you care about, and returns JSON that validates against your schema — reliably enough that a downstream system can consume it without a human in the loop. The through-line is a promise you’re making to that downstream system: every record will match the schema, and a missing value will be an honest null, never an invention. Each stage builds one part of keeping that promise. You are not doing seven disconnected exercises; you are building one extraction pipeline in seven passes.

The one thing you’re building

A structured data extraction pipeline. It reads an unstructured document, uses a constrained-output call against a JSON Schema you design, validates the result, retries on failure, and hands a clean, schema-valid record to a downstream system — handling missing and ambiguous data gracefully rather than guessing.

input → an unstructured document output → a schema-valid JSON record promise → valid every time, no fabrication

How the seven stages connect

Read the diagram as a document’s journey to a validated record. A schema (Stage 1) defines the shape; constrained decoding (Stage 2) forces the model’s output into it; the prompt (Stage 3) and edge-case handling (Stage 4) make the contents accurate and honest; the validate-and-retry loop (Stage 5) is the gate that guarantees correctness the schema alone can’t; scaling (Stage 6) runs it over many documents; and the downstream contract (Stage 7) is who consumes the result.

Don’t worry if a box uses a word you don’t know yet — the glossary below defines every term in the diagram, and each stage explains its own box in full when you reach it.

The terms this lab uses

These come up across multiple stages. Skim them now; you don’t need to memorize them — they’ll make sense as you build.

Structured output
Getting the model to return data in a shape you specify — here, JSON matching a schema — rather than free text. The whole point of an extraction system: output a program can consume, not prose a human has to parse.
JSON Schema
The formal description of your target shape: which fields exist, their types, which are required. It’s both the instruction to the model and the yardstick you validate against. Design it well and everything downstream is easier.
Constrained decoding
The mechanism behind guaranteed-valid output: the API compiles your schema into a grammar and restricts token generation so the response must fit the shape. It guarantees the format — not that the values are correct.
output_config.format (JSON outputs)
The request field that constrains the model’s final response to your JSON Schema ({"type":"json_schema","schema":{…}}). The primary tool when the whole reply should be one structured record.
Strict tool use
The other constraint path: mark a tool’s input schema strict: true so the model’s tool call arguments are guaranteed to match. Right when extraction feeds a tool/function rather than being the whole reply.
Schema subset
Constrained decoding supports most, not all, of JSON Schema: every object needs additionalProperties: false and all keys in required; no recursion; some constraints (min/max, string length) aren’t enforced and must be post-validated.
Nullable / optional field
How you let the model say “not present” honestly: a nullable type (["string","null"]) or omitting a field from required. The alternative — forcing a value — is what causes fabrication.
Enum + “other”
An escape hatch for classification fields: include an "other" (or "unknown") option in an enum so the model isn’t forced to miscategorize a value that doesn’t fit any listed category.
Fabrication / hallucination
The core reliability risk in extraction: the model inventing a plausible value for something the document doesn’t actually contain. Most edge-case design exists to prevent exactly this.
Validation & retry
The reliability loop: check the output against the schema (and your business rules), and on failure feed the error back for another attempt. Necessary because constrained decoding guarantees shape, not accuracy or rule-compliance.
Batch (Message Batches API)
The path for bulk extraction: submit many documents in one batch, processed asynchronously at lower cost — for when you’re extracting from thousands of files, not one interactively.
Downstream contract
The agreement with whatever consumes your records — a database, a queue, another service. The schema is that contract; honoring it (valid shape, honest nulls, stable fields) is what lets the next system trust your output.
Before Stage 1
  • You can state, in one sentence, what the finished pipeline promises its downstream consumer.
  • You can trace a document through the diagram: schema + doc → constrained call → validate/retry → downstream.
  • You can explain why “guaranteed-valid format” is not the same as “correct data.”
Stage 01

Design the extraction schema

Prompt Engineering & Structured Output · 20%

The schema is the foundation of the whole system — it’s the target shape, the instruction to the model, and the yardstick you validate against, all at once. A well-designed schema makes extraction accurate and downstream integration trivial; a sloppy one causes fabrication and breakage. Start here, deliberately.

You’re extracting from invoices: vendor, invoice number, date, line items, total. Some invoices have a PO number; some don’t. Some totals are ambiguous. The schema has to express all of that precisely — what’s required, what’s optional, what can be absent.

1a · Write the target schema

Describe the record you want. Field names and descriptions aren’t decoration — the model reads them as extraction instructions, so name them well.

{ "type": "object", "properties": { "vendor_name": { "type": "string", "description": "Legal name of the issuing vendor" }, "invoice_number": { "type": "string" }, "invoice_date": { "type": "string", "description": "ISO 8601 date, YYYY-MM-DD" }, "po_number": { "type": ["string", "null"], "description": "PO number, or null if absent" }, "total": { "type": "number" } }, "required": ["vendor_name", "invoice_number", "invoice_date", "po_number", "total"], "additionalProperties": false }

1b · Respect the supported subset

Constrained decoding supports most of JSON Schema, but not all — and the exam expects you to know the boundaries:

  • Every object needs additionalProperties: false, and every property must be listed in required. (To make a field “optional,” you keep it required but allow null — see 1c.)
  • No recursive schemas — a schema that references itself won’t compile. Flatten nested/recursive structures.
  • Some constraints aren’t enforced by decoding — numeric minimum/maximum, string minLength, regex pattern. The shape is guaranteed; those value-level rules you must post-validate (Stage 5).
  • Enums are supported and are your friend for classification fields.

1c · Model “might be absent” explicitly

The single most important schema-design decision for accuracy: give the model a legitimate way to say “this isn’t here.” A field that might be absent gets a nullable type (["string","null"]), so the model returns null instead of inventing a value to satisfy a required string. This one choice prevents a whole class of fabrication, and Stage 4 builds on it.

Exam lens Know the supported subset cold: additionalProperties: false everywhere, all keys required, no recursion, and that min/max/length/pattern aren’t enforced by decoding. Know that descriptions guide extraction and that nullable types are how you avoid forcing fabrication. “Just mark the field optional” is a distractor — the subset wants it required-and-nullable.
Verify The exact supported JSON Schema subset (recursion, $ref, which constraints are enforced) is evolving as structured outputs mature. Confirm the current subset against platform.claude.com/docs before finalizing a schema.
Stage 1 done when
  • You have a schema for your document type with typed fields and clear descriptions.
  • It obeys the subset: additionalProperties: false, all keys required, no recursion.
  • Fields that may be absent are nullable, not silently forced.
Stage 02

Constrain the output

Prompt Engineering & Structured Output · 20%

With a schema in hand, you make the model’s output conform to it — not by asking nicely, but by constraining generation so the response must match. There are two ways to do this, and knowing which fits your case is a core exam skill.

2a · JSON outputs: constrain the final response

When the whole reply should be one structured record, pass your schema in output_config.format. The API compiles it to a grammar and constrains decoding, so the response is guaranteed to fit the schema.

import anthropic client = anthropic.Anthropic() msg = client.messages.create( model="claude-opus-4-8", max_tokens=1024, messages=[{"role": "user", "content": f"Extract the invoice fields:\n\n{document}"}], output_config={ "format": { "type": "json_schema", "schema": invoice_schema, # the Stage 1 schema } }, ) # msg.content is guaranteed to parse and match the schema shape
Structured outputs recently reached general availability. Earlier code used a beta header (structured-outputs-2025-11-13) and/or an output_format field; the current surface is output_config.format. If you’re following an older tutorial, that’s the rename to watch for.

2b · Strict tool use: constrain tool inputs

The other path fits when extraction feeds a tool (a function you’ve defined) rather than being the whole reply. Mark the tool’s input schema strict: true, and the model’s tool-call arguments are guaranteed to match it. Same guarantee, applied to the tool’s inputs instead of the final message.

  • Use JSON outputs when the response is the record.
  • Use strict tool use when the record is an argument to a function the model calls — and you can combine both in one request.

2c · What the guarantee does and doesn’t cover

Constrained decoding is powerful but narrow: it guarantees the output parses and matches the shape. It does not guarantee the values are right, and it doesn’t enforce the constraints the subset excludes (min/max, pattern). This is the hinge the next three stages turn on — format is solved here; accuracy and rules are not.

Exam lens Know the two constraint modes and when each applies — output_config.format for the final response, strict: true tool use for tool arguments — and that they combine. Know that constrained decoding is constrained decoding / grammar-based, guaranteeing shape only. The older “force a tool with tool_choice and hope” pattern is worth recognizing as the predecessor.
Verify output_config.format, the strict tool flag, whether any beta header applies, and incompatibilities (e.g. citations) are current but settling post-GA. Confirm against platform.claude.com/docs.
Stage 2 done when
  • Your call returns JSON that parses and matches the schema, every time, via constrained output.
  • You can state whether your case wants JSON outputs or strict tool use, and why.
  • You can explain, in one sentence, what the guarantee does not cover.
Stage 03

Prompt for accurate extraction

Prompt Engineering & Structured Output · 20%

The schema guarantees shape; the prompt is what makes the values right. A constrained call with a lazy prompt gives you perfectly-formatted wrong answers. This stage is about the prompt-engineering that lifts accuracy inside the guaranteed shape.

The schema forces a total number field — so the model will always return a number. Whether it returns the right number (the grand total, not the subtotal) is entirely down to how you prompt and describe the field.

3a · Let the field descriptions do work

The descriptions you wrote in the schema are live extraction instructions. “total: the grand total including tax, not the subtotal” disambiguates a field far more reliably than hoping the model picks the right number. Precise descriptions are the highest-leverage accuracy tool you have, because they travel with the constraint.

3b · Give an example, and ground the extraction

Two techniques that raise accuracy on messy documents:

  • A worked example in the prompt — a small sample document and its correct extraction — shows the model exactly how to handle your format’s quirks.
  • Ask it to ground values in the source: instruct the model to extract only what’s present and, where useful, to quote the span it pulled a value from. Extraction tied to the text is far less prone to drift than extraction from memory.

3c · Tell it what to do when unsure

The instruction that most reduces fabrication: state the rule explicitly — “if a field is not present in the document, return null; do not guess or infer a plausible value.” The schema’s nullable types (Stage 1) give the model somewhere honest to put “I don’t know”; this instruction tells it to use them. Together they turn “absent” from a fabrication trigger into a clean null.

Exam lens Know that format guarantees don’t buy accuracy — the prompt does. The reliable levers: rich field descriptions, a worked example, grounding values in the source text, and an explicit “return null when unsure, don’t guess” instruction. Scenarios that show a well-formed but wrong extraction are pointing at a prompt problem, not a schema problem.
Stage 3 done when
  • Field descriptions disambiguate the tricky fields (which total, which date).
  • Your prompt includes a worked example and asks the model to ground values in the text.
  • It explicitly instructs “null when absent, never guess” — and you’ve seen it comply.
Stage 04

Handle edge cases gracefully

Context Management & Reliability · 15%

Real documents are messy: fields missing, values ambiguous, categories that don’t fit your enum, whole sections absent. A brittle extractor fabricates or crashes on these; a graceful one has a defined, honest answer for each. This is where extraction reliability is won or lost.

One invoice has no PO number. Another lists a currency you didn’t anticipate. A third is missing its date entirely. The system has to produce a valid record for each — without inventing a PO, mis-slotting the currency, or refusing outright.

4a · Missing fields become honest nulls

This is the payoff of the Stage 1 nullable types plus the Stage 3 instruction: a missing field returns null, the record still validates, and the downstream system gets an honest “not present” instead of a fabricated value. Missing data is a normal, valid outcome — not an error and not an invention.

4b · The enum + “other” escape hatch

For classification fields, a closed enum forces the model to pick one of your listed values — so an unanticipated category gets crammed into the nearest wrong bucket. The fix is an escape hatch: include "other" (or "unknown") in the enum. Now a value that fits nothing lands in other honestly, instead of a confident miscategorization.

"currency": { "type": "string", "enum": ["USD", "EUR", "GBP", "INR", "other"] # escape hatch prevents miscategorization }

4c · Add a confidence or notes field

For genuinely ambiguous extractions, give the model a place to flag uncertainty — a nullable extraction_notes field, or a per-field confidence. A record that says “total was ambiguous: two candidate figures” is far more useful downstream than a silent guess, because the consumer can route low-confidence records to human review. Graceful handling means surfacing uncertainty, not hiding it inside a clean-looking value.

Exam lens The reliability core of this scenario. Know the three moves: nullable for missing (honest null, not fabrication), enum + “other” (no forced miscategorization), and a notes/confidence field to surface ambiguity for downstream routing. “Make every field required so nothing is missing” is the trap — it forces fabrication, the opposite of graceful.
Verify Enum support and nullable-type handling under constrained decoding are current, but confirm the exact enum/nullable behavior against platform.claude.com/docs as the subset settles.
Stage 4 done when
  • A document with a missing field yields a valid record with an honest null.
  • An unanticipated category lands in other, not a wrong bin.
  • Ambiguous extractions surface their uncertainty instead of hiding it.
Stage 05

Validate & retry

Context Management & Reliability · 15%

Constrained decoding guarantees the shape — but not that the values obey your business rules, and not the constraints the subset doesn’t enforce. So you validate anyway, and on failure you retry with the error. This loop is what turns “usually right” into “reliably right.”

The schema guarantees total is a number. It does not guarantee the line items sum to that total, that the date is in the past, or that the tax rate is between 0 and 1. Those are your rules — and only a validation step enforces them.

5a · Validate the shape you were promised anyway

Even with constrained output, validate against your schema with a real validator (jsonschema, ajv). It’s cheap insurance against the edges — a truncated response (hit max_tokens), a refusal, an API change — and it’s the boundary check a reliable pipeline always keeps.

5b · Then validate the rules decoding can’t

This is the part the schema guarantee never covered: the value-level and cross-field rules. Numeric ranges, string formats, and regex patterns (excluded from the enforced subset), plus business logic like “line items sum to total” or “date is not in the future.” Run these as a validation layer after parsing.

from jsonschema import validate, ValidationError def check(record): validate(record, invoice_schema) # shape + enforced subset assert 0 <= record["tax_rate"] <= 1, "tax_rate out of range" assert sum(li["amount"] for li in record["line_items"]) == record["total"], "lines != total"

5c · Retry with the error fed back

When validation fails, don’t just re-run the same call — retry with the specific error added to the prompt (“the previous extraction had line items summing to 90 but a total of 100; re-extract and resolve”). The model corrects far more reliably when it knows what was wrong. Bound the retries (two or three), and if it still fails, route the document to human review rather than looping forever or shipping a bad record.

Exam lens The defining idea of this scenario’s reliability domain: constrained output guarantees format, not correctness — so you still validate, especially the value-level and business rules decoding doesn’t enforce. Know the validate → retry-with-error → bounded fallback loop, and that “constrained decoding means I don’t need to validate” is the trap answer. Also know stop_reason catches truncation/refusal.
Verify Which constraints constrained decoding enforces vs. leaves to you shifts as the feature matures — re-confirm the enforced subset so your validation layer covers exactly the gaps. Check platform.claude.com/docs.
Stage 5 done when
  • Every record is schema-validated after parsing, even though output is constrained.
  • Your business rules (ranges, cross-field sums, formats) are checked in a validation layer.
  • Failures retry with the specific error, bounded, then fall back to review.
Stage 06

Scale to many documents

Context Management & Reliability · 15%

Extracting one document interactively is the easy case. Real systems process thousands — and that changes what matters: cost, throughput, long documents that strain context, and the failure modes that only show up at volume.

6a · Batch for bulk, async extraction

When you’re extracting from many documents and don’t need each answer instantly, the Message Batches API is the right tool: submit a large set of extraction requests in one batch, processed asynchronously at lower cost than one-by-one interactive calls. It’s built for exactly this bulk-extraction shape — not for the interactive single-document case.

6b · Long documents and the context window

A single document can be large enough to strain the context window — a 90-page contract, a scanned report. When the source is too big to extract in one pass reliably, split it: extract per-section against the same schema and merge, or first locate the relevant section (retrieval) and extract from that. Keeping the working context focused on the relevant text improves both accuracy and cost, the same principle that governs any long-context task.

6c · Handle refusals and truncation at volume

At scale, rare failure modes become certainties — some document trips a refusal, some response truncates at max_tokens. Check stop_reason on every response: a max_tokens stop means the record is incomplete (raise the limit or split), and a refusal means that document needs a different path. At volume you handle these programmatically, not by eyeballing, so one bad document doesn’t silently corrupt the batch.

Exam lens Know the Message Batches API is for bulk/async extraction (cost, throughput), not interactive single calls. Know that long documents are a context problem — split or retrieve-then-extract. And know to monitor stop_reason for truncation and refusals at volume. “Send all 10,000 docs as interactive calls” is the wrong-tool distractor.
Verify Batch API limits (requests per batch, turnaround, discount), and whether structured outputs apply cleanly in batch, are current but evolving. Confirm against platform.claude.com/docs.
Stage 6 done when
  • Bulk extraction runs through the Batches API rather than a loop of interactive calls.
  • A document too large for one pass is split (or retrieved) and merged correctly.
  • stop_reason is checked so truncations and refusals are handled, not silently dropped.
Stage 07

Integrate downstream

Integration in practice

Extraction isn’t the goal — feeding a downstream system is. A database, a queue, another service consumes your records, and the whole pipeline’s value depends on that handoff being clean and dependable. The schema you designed in Stage 1 turns out to be the contract that makes it work.

7a · The schema is the contract

Whatever consumes your output — a table, an API, a message queue — expects a specific shape. Your JSON Schema is that agreement: honor it and the downstream system can rely on every record without defensive parsing. This is why the earlier stages matter downstream: constrained output keeps the shape, validation keeps the rules, honest nulls keep the semantics. The contract holds end to end.

7b · Decide what happens to invalid records

A production pipeline needs a defined path for the records that don’t make it through — the ones that failed validation after retries. Route them somewhere explicit: a dead-letter queue, a review table, an alert. What you must not do is silently drop them or let them through unvalidated. A clean pipeline is one where every document ends in a known state: extracted-and-valid, or flagged-for-review.

7c · Version the schema as it evolves

The contract will change — a new field, a tightened type. Version it, so downstream consumers aren’t broken by a silent shift and can migrate deliberately. Treat the extraction schema the way you’d treat any API contract between two systems, because that’s exactly what it is.

Exam lens The integration framing: the schema is the contract with the downstream system, which is why shape + validation + honest nulls matter beyond the extraction itself. Know that invalid records need an explicit path (dead-letter/review), never a silent drop, and that a schema is versioned like any API contract. Reliability here means every document ends in a known state.
Stage 7 done when
  • Valid records flow to the downstream system in the exact schema shape it expects.
  • Invalid records go to an explicit review/dead-letter path — never silently dropped.
  • The schema is versioned so downstream consumers migrate deliberately.
Wrap-up

What you built, and how it maps

You now have one working extraction pipeline that exercises the scenario’s two primary domains: Prompt Engineering & Structured Output (20%) — schema design within the supported subset, constrained output via output_config.format and strict tool use, and the prompting that makes values accurate inside the guaranteed shape — and Context Management & Reliability (15%) — graceful edge-case handling, the validate-and-retry loop, scaling with batch and long-document strategies, and a clean downstream contract. Completing it, you should be able to answer any question the scenario raises — what the format guarantee does and doesn’t cover, why nullable beats forced-required, when to validate despite constrained output, and how the schema serves as the downstream contract.

The exam is 60 scenario questions in 120 minutes, and it rewards judgment over recall — the wrong answers are written to tempt people with partial knowledge. Having actually designed the schema, watched constrained output guarantee shape but not accuracy, and built the validation loop that closes that gap is the difference between recognizing the right call and guessing.
