Tool Design & MCP Integration

Section 2.1

Design effective tool interfaces with clear descriptions and boundaries

Task Statement 2.1

A tool’s description isn’t documentation for you — it’s the primary signal the model uses to decide whether to call that tool. Two tools with near-identical descriptions are two tools the model will confuse. Most tool-selection bugs are description bugs.

What & why

The LLM selects tools by reading their descriptions, so a minimal or vague description leads to unreliable selection — especially among similar tools. A good description states the tool’s purpose, expected inputs and outputs, example queries, edge cases, and crucially when to use it versus a similar alternative. Overlapping descriptions (analyze_content vs analyze_document with near-identical text) cause misrouting. And it’s not just the tool descriptions: keyword-sensitive wording in the system prompt can create unintended tool associations that override even a well-written description.

Knowledge points
  • Tool descriptions are the primary mechanism for tool selection; minimal descriptions cause unreliable choices among similar tools.
  • Good descriptions include input formats, example queries, edge cases, and boundary explanations.
  • Ambiguous or overlapping descriptions cause misrouting (the analyze_content vs analyze_document problem).
  • System-prompt wording affects selection — keyword-sensitive instructions can create unintended tool associations.
In practice
  • Write descriptions that differentiate purpose, inputs, outputs, and when-to-use vs alternatives.
  • Rename and re-describe to eliminate overlap — e.g., rename analyze_content to extract_web_results with a web-specific description.
  • Split generic tools into purpose-specific ones with defined I/O contracts — a generic analyze_document becomes extract_data_points, summarize_content, and verify_claim_against_source.
  • Review system prompts for keyword-sensitive instructions that might override good tool descriptions.
Anti-patterns the exam tests
  • Minimal descriptions (“analyzes documents”) that give the model nothing to discriminate on.
  • Two tools with overlapping descriptions — the model picks unpredictably between them; the fix is renaming/splitting, not prompt-nudging.
  • Blaming the model for misrouting when the real cause is a system-prompt keyword steering it, or descriptions that don’t say when-to-use-which.
  • One generic mega-tool doing several jobs, when purpose-specific tools with clear contracts would select more reliably.
Exam lens A scenario shows the model calling the wrong one of two similar tools. The tempting answer is “add examples to the prompt” or “tell the model which to use.” The stronger answer fixes the tool interfaces: rename to remove overlap, sharpen the descriptions to say when each applies, or split a generic tool into purpose-specific ones. Also check the system prompt for a keyword quietly biasing selection.

Section 2.2

Implement structured error responses for MCP tools

Task Statement 2.2

When a tool fails, the agent’s next move depends entirely on how it failed. A generic “Operation failed” tells the agent nothing, so it can’t recover intelligently. Structured errors — category, retryable, a human-readable reason — are what let the agent make the right recovery decision.

What & why

MCP communicates tool failures back with the isError flag, but the flag alone isn’t enough — the agent needs to know what kind of failure it was. There are four categories: transient (timeouts, service unavailable), validation (invalid input), business (policy violation), and permission errors. Uniform “Operation failed” responses prevent appropriate recovery — the agent can’t tell a retryable timeout from a permanent policy block, so it either wastes retries or gives up when it shouldn’t. Returning structured metadata (category, an isRetryable boolean, a readable description) is what enables the right decision.

Knowledge points
  • The MCP isError flag pattern for communicating failures back to the agent.
  • The four error types: transient (timeouts, unavailability), validation (bad input), business (policy), permission.
  • Why uniform “Operation failed” responses prevent appropriate recovery.
  • The retryable vs non-retryable distinction, and how structured metadata prevents wasted retry attempts.
In practice
  • Return structured error metadata: errorCategory (transient/validation/permission), an isRetryable boolean, and human-readable descriptions.
  • Include retriable: false and a customer-friendly explanation for business-rule violations, so the agent can communicate appropriately rather than retry.
  • Implement local recovery within subagents for transient failures, propagating to the coordinator only what can’t be resolved locally — along with partial results and what was attempted.
  • Distinguish access failures (needing retry decisions) from valid empty results (a successful query that simply found no matches).
Anti-patterns the exam tests
  • Returning a generic “Operation failed” — the agent can’t choose a recovery path.
  • Making a business-rule violation look retryable — the agent burns retries on something that will never succeed.
  • Treating a valid empty result as an error (or an error as an empty result) — conflates “couldn’t look” with “looked and found nothing.”
  • Propagating every transient blip to the coordinator instead of recovering locally first.
Exam lens A scenario shows an agent retrying a refund that’s blocked by policy, or giving up on a transient timeout. The fix is structured error responses: mark the policy block retriable: false with a clear reason, mark the timeout transient/retryable. Watch for the empty-result trap — “no orders found” is a successful query, not a failure, and answers that treat it as an error are wrong.
⚑

Check your understanding
  • Name the four error categories and which are typically retryable.
  • Why is “no matches found” not an error?
  • What should a subagent do with a transient failure before involving the coordinator?
Section 2.3

Distribute tools appropriately across agents and configure tool choice

Task Statement 2.3

More tools does not mean more capable. Every extra tool an agent can see adds to the decision it has to make each turn, and past a handful, selection reliability drops. Scope each agent to the tools its role actually needs — and use tool_choice to control when a tool must be called.

What & why

Giving an agent too many tools (18 instead of 4–5) degrades selection reliability by increasing decision complexity — more options, more chances to pick wrong. Agents also tend to misuse tools outside their specialization (a synthesis agent attempting web searches it shouldn’t). The fix is scoped access: each agent gets only the tools its role needs, with limited cross-role tools for specific high-frequency needs. Separately, tool_choice controls the calling behavior: "auto" (model may return text instead of a tool), "any" (must call a tool, its choice), or forced ({"type":"tool","name":"..."}, must call a specific one).

Knowledge points
  • Too many tools degrades selection — 18 vs 4–5 increases decision complexity and error rate.
  • Agents misuse tools outside their specialization (a synthesis agent doing web searches).
  • Scoped tool access: only the tools a role needs, with limited cross-role tools for high-frequency needs.
  • tool_choice options: "auto", "any", and forced {"type":"tool","name":"..."}.
In practice
  • Restrict each subagent’s tool set to its role to prevent cross-specialization misuse.
  • Replace generic tools with constrained alternatives — swap fetch_url for load_document that validates document URLs.
  • Provide scoped cross-role tools for high-frequency needs (a verify_fact tool for the synthesis agent) while routing complex cases through the coordinator.
  • Use forced tool_choice to run a specific tool first (force extract_metadata before enrichment), then continue in follow-up turns.
  • Set tool_choice: "any" to guarantee a tool call rather than conversational text.
Anti-patterns the exam tests
  • Giving every agent every tool “for flexibility” — degrades selection for all of them.
  • Letting a specialist keep tools outside its role — it will eventually misuse them.
  • Using a broad generic tool (fetch_url) where a validated, constrained one (load_document) would be safer and clearer.
  • Relying on the prompt to force a first tool when forced tool_choice guarantees it.
Exam lens A scenario shows an agent with a large toolset picking wrong, or a synthesis agent wandering into search. The answer is scope the tools to the role (4–5, not 18), not “improve the descriptions” alone. When the ask is “guarantee a tool runs” or “run this one first,” the answer is tool_choice — "any" to force some tool, forced-name to pin a specific one.
⚑
Build it in a capstone The Multi-Agent Research System lab scopes each subagent’s tools to its role (searcher has web tools, synthesizer has none), and the Developer Productivity lab widens tools deliberately rather than granting all at once.
Check your understanding
  • Why does an 18-tool agent select less reliably than a 4-tool one?
  • When would you force a specific tool vs set tool_choice: "any"?
  • What’s the argument for load_document over a generic fetch_url?
Section 2.4

Integrate MCP servers into Claude Code and agent workflows

Task Statement 2.4

MCP is how agents reach beyond built-in tools into your systems — issue trackers, databases, internal APIs. Getting integration right is about scoping (who shares a server), credentials (without committing secrets), and discoverability (helping the agent find and prefer the right tools).

What & why

MCP servers are scoped two ways: project-level (.mcp.json, committed, for shared team tooling) and user-level (~/.claude.json, for personal or experimental servers). Credentials go in via environment-variable expansion (${GITHUB_TOKEN}) so secrets aren’t committed. Tools from all configured servers are discovered at connection time and available simultaneously. And MCP resources expose content catalogs (issue summaries, doc hierarchies, database schemas) so the agent can see what data exists without spending exploratory tool calls to find out.

Knowledge points
  • Scoping: project-level .mcp.json (shared team tooling) vs user-level ~/.claude.json (personal/experimental).
  • Environment-variable expansion in .mcp.json (${GITHUB_TOKEN}) for credentials without committing secrets.
  • Tools from all configured servers are discovered at connection time and available simultaneously.
  • MCP resources expose content catalogs to reduce exploratory tool calls.
In practice
  • Configure shared servers in project-scoped .mcp.json with env-var expansion for auth tokens.
  • Configure personal/experimental servers in user-scoped ~/.claude.json.
  • Enhance MCP tool descriptions to explain capabilities and outputs in detail — otherwise the agent may prefer a built-in like Grep over a more capable MCP tool.
  • Prefer existing community MCP servers (e.g., Jira) over custom builds for standard integrations; reserve custom servers for team-specific workflows.
  • Expose content catalogs as MCP resources so agents see available data without exploratory calls.
Anti-patterns the exam tests
  • Putting a shared team server in user-level config (or a personal one in the committed project config) — wrong scope, teammates don’t get it (or get it unwanted).
  • Committing secrets instead of using ${ENV_VAR} expansion.
  • Thin MCP tool descriptions that lose out to built-in tools the agent already knows.
  • Building a custom server for a standard integration a community server already covers.
  • Forcing the agent to explore for data that a resource catalog could expose directly.
Exam lens A scenario asks where a team’s shared Jira server belongs, or why the agent keeps using Grep instead of a better MCP tool. Scope answers: shared → project .mcp.json; personal → ~/.claude.json. The “prefers Grep” problem is a weak tool description, not a model flaw. Secrets in config is always the wrong answer — ${ENV_VAR} is the right one.
⚑
Build it in a capstone The Developer Productivity lab integrates an MCP server (external via command/URL vs in-process), scopes its tools, and writes descriptions strong enough that the agent reaches for it — this section in practice.
Check your understanding
  • Which config file is shared via version control, and which is personal?
  • How do credentials get into .mcp.json without committing secrets?
  • Why might an agent prefer Grep over a more capable MCP tool, and how do you fix it?
Section 2.5

Select and apply built-in tools effectively

Task Statement 2.5

The built-in tools — Read, Write, Edit, Bash, Grep, Glob — are the agent’s hands in a codebase. Using them well is knowing which fits which job: Grep for content, Glob for paths, Edit for targeted changes, and the Read+Write fallback when Edit can’t find a unique anchor.

What & why

Each built-in has a job. Grep searches file contents for patterns — function names, error messages, import statements. Glob matches file paths by name or extension pattern. Read/Write do full-file operations; Edit makes targeted modifications by matching unique text — and when that text isn’t unique, Edit fails, so you fall back to Read + Write for a reliable change. The larger skill is incremental exploration: start with Grep to find entry points, then Read to follow imports and trace flows, rather than reading everything up front.

Knowledge points
  • Grep for content search (patterns like function names, error messages, imports).
  • Glob for file-path pattern matching (files by name or extension).
  • Read/Write for full-file operations; Edit for targeted changes via unique text matching.
  • When Edit fails on non-unique matches, use Read + Write as the fallback.
In practice
  • Grep to search code content across a codebase (all callers of a function, a specific error message).
  • Glob to find files by naming pattern (**/*.test.tsx).
  • Read then Write when Edit can’t find unique anchor text.
  • Build understanding incrementally: Grep for entry points, then Read to follow imports and trace flows — not reading all files upfront.
  • Trace usage across wrapper modules: identify all exported names, then search each across the codebase.
Anti-patterns the exam tests
  • Using Glob when you mean Grep (or vice versa) — path pattern vs content search are different jobs.
  • Reading every file upfront to “understand the codebase” instead of Grep-then-Read incrementally — floods context (ties to Module 5).
  • Retrying Edit repeatedly on non-unique text instead of falling back to Read + Write.
Exam lens A scenario asks which tool for a task: “find every caller of processPayment” → Grep; “list all .test.tsx files” → Glob; “Edit keeps failing because the text appears three times” → Read + Write. The “read the whole repo first” option is the context-flooding distractor — incremental Grep-then-Read is the expected approach.

Module 2 · Wrap-up

What this domain comes down to

Tool Design & MCP Integration is about the surface between the agent and everything it can do: descriptions the model selects on (2.1), errors structured so the agent can recover (2.2), tools scoped to each role with tool_choice controlling when they fire (2.3), MCP servers scoped and credentialed correctly with discoverable tools (2.4), and the built-ins used for the jobs they fit (2.5).

Two threads run through the anti-patterns. First: the model acts on what you give it — descriptions and error structure — not on what you meant. Vague descriptions misroute; generic errors block recovery; thin MCP descriptions lose to built-ins. Second: less is more reliable — fewer, role-scoped tools select better than a big generic pile. Fix the interface, scope the access, and most of this domain’s scenarios resolve.
