Before you start

You should be comfortable in a terminal and working in a real codebase — this lab assumes the developer experience the exam itself assumes. It does not stop to explain git or what a build command is.

  • Claude Code installed and authenticated (claude auth status should show you logged in). The native installer is the current recommended path.
  • A real repo you know well — ideally one with a build, a test suite, and a few modules. You’ll configure Claude Code on this repo, so pick something you can commit config to.
  • A few minutes in the CLI to try the Shift+Tab permission-mode cycle and the built-in commands (/help, /context, /compact) before Stage 4.
Verify before you build Claude Code moves fast — command locations (.claude/commands/ vs. the newer .claude/skills/), frontmatter keys, the permission-mode cycle, and key bindings all carry real shelf-life. Wherever you see a Verify note, check it against the current docs at code.claude.com/docs. The shape of each pattern is stable; exact identifiers and shortcuts drift between versions.
Orientation

What you’re building, before you build it

By the end of this lab you’ll have one thing set up: a real repository with a complete, committed Claude Code workflow on it — project context that every teammate inherits, four slash commands that turn your team’s daily tasks into six keystrokes, a clear rule for when to plan and when to execute, and guardrails that hold even when someone misconfigures their session. Every stage configures one more piece of that same setup. You are not doing seven disconnected exercises; you are outfitting one repo in seven passes.

The one thing you’re building

A team-ready Claude Code workflow on a repo you know. It carries your conventions in a layered CLAUDE.md hierarchy, exposes /generate, /refactor, /debug, and /document as shared commands, encodes when to use plan mode versus direct execution, and stays reliable across long sessions and multiple teammates.

input → a repo + your conventions output → a committed .claude/ setup payoff → the whole team inherits it

How the seven stages connect

Read the diagram as a configured workspace. At the center is a Claude Code session working on your repo. The CLAUDE.md hierarchy (Stages 1–2) feeds it context; the four slash commands (Stage 3) are the actions your team invokes; the plan-vs-execute decision (Stage 4) governs how each task runs; context management (Stage 5) keeps long sessions healthy; guardrails (Stage 6) enforce what must always hold; and shipping (Stage 7) commits the whole setup so the team inherits it.

Don’t worry if a box uses a word you don’t know yet — the glossary below defines every term in the diagram, and each stage explains its own box in full when you reach it.

The terms this lab uses

These come up across multiple stages. Skim them now; you don’t need to memorize them — they’ll make sense as you build.

Claude Code
Anthropic’s command-line coding tool — you run claude in a repo and it reads, writes, and runs commands to help you develop. This lab is about configuring and driving it (not building an agent with the SDK).
CLAUDE.md
A markdown file of persistent instructions Claude Code loads at the start of every session — your build commands, conventions, and architectural notes, so you don’t re-explain them each time. Treated as context, not enforced rules.
Configuration hierarchy
The layered set of memory files — user (~/.claude/CLAUDE.md), project (./CLAUDE.md), path-scoped rules (.claude/rules/), and more — where more specific instructions take precedence over broader ones.
Path-scoped rule
A file in .claude/rules/ with a paths: glob in its frontmatter, so its instructions load only when Claude works on matching files. How you keep area-specific rules out of every session’s context.
Slash command
A reusable prompt you invoke by typing /name. Defined as a markdown file in .claude/commands/ (legacy) or .claude/skills/<name>/SKILL.md (current), with optional frontmatter (description, allowed-tools, argument-hint, model) and a $ARGUMENTS placeholder.
Plan mode
A read-only permission mode where Claude Code investigates and proposes a plan but changes nothing until you approve. Reached with Shift+Tab, /plan, or --permission-mode plan. Its read-only nature is enforced at the tool level.
Direct execution
Letting Claude Code make changes as it goes — the default and acceptEdits modes — without a separate planning step. Right for small, well-scoped, reversible tasks.
Permission mode
How often Claude Code pauses for approval: default (asks), acceptEdits (auto-approves edits), plan (read-only), plus dontAsk / bypass for automation. Cycle the common three with Shift+Tab.
Context (window)
Everything the session holds at once — CLAUDE.md, files read, the conversation. It’s finite; a long session fills it and adherence drops. /context shows usage; /compact compresses it.
Compaction
Compressing the conversation to free context (/compact, or automatic near the limit). Project-root CLAUDE.md survives it — re-read from disk — but conversation-only instructions do not.
Hook
A callback that fires at a lifecycle point (e.g. PreToolUse) and can allow, deny, or modify an action. Where hard guarantees belong, because a hook’s deny holds regardless of what Claude decides — unlike CLAUDE.md, which is guidance.
Explore → Plan → Execute
The workflow this lab teaches: let Claude explore the code, produce a plan you approve, then switch to execution. The pattern that makes plan mode pay off on non-trivial work.
Before Stage 1
  • You can state, in one sentence, what the finished setup gives your team.
  • You can trace a task through the diagram: context + command → session → plan or execute.
  • You can name the difference between CLAUDE.md (guidance) and a hook (enforced).
Stage 01

Pick the repo & write CLAUDE.md

Claude Code Configuration & Workflows · 20%

Everything starts with project context. A good CLAUDE.md is the onboarding document Claude reads before every session — it’s the highest-leverage file in your whole setup, because a bad line here degrades every task. Write a tight one for your repo.

Every new session, Claude Code starts knowing nothing about your repo. Without CLAUDE.md, you re-explain your build command, your conventions, and your architecture every single time.

1a · Create the project CLAUDE.md

It lives at ./CLAUDE.md (or ./.claude/CLAUDE.md) in your repo root, and it’s committed — shared with everyone on the team. Run claude in your repo and use the /init command to scaffold one from your codebase, then edit it down.

./CLAUDE.md
# [Your Project] — context for Claude Code ## Commands - build: `make build` - test: `pytest -q` - lint: `ruff check .` ## Conventions - Python 3.11, type hints required on public functions. - Errors: raise typed exceptions, never return None on failure. - Tests live next to the module as `test_*.py`. ## Architecture - `api/` HTTP layer · `core/` domain logic · `store/` persistence. - Never import `store/` directly from `api/` — go through `core/`.

1b · What belongs — and what doesn’t

The discipline that separates a useful CLAUDE.md from a bloated one:

  • Do include: build/test/lint commands, project-wide conventions, architectural constraints, naming rules — things that apply to every session and that Claude can’t infer from the code.
  • Don’t include: generic advice (“write clean code” — Claude already knows), one-off task instructions (those go in the prompt), or detailed style rules a linter should enforce. Claude is an in-context learner; if your code is consistent, it picks up style by reading it.

1c · Keep it under ~200 lines

CLAUDE.md loads into context at the start of every session, spending tokens alongside your work. The docs target under 200 lines; past that, context fills and adherence to your instructions actually drops. If it’s growing, that’s the signal to split into path-scoped rules — Stage 2.

Exam lens Know that CLAUDE.md is context, not enforcement — Claude treats it as guidance, so “put a hard rule in CLAUDE.md” is the wrong answer when the scenario needs a guarantee (that’s a hook, Stage 6). Know the 200-line adherence effect, and that style belongs in a linter, not CLAUDE.md. These are recurring distractors.
Verify /init, the ./CLAUDE.md vs ./.claude/CLAUDE.md locations, and the 200-line guidance are current, but specifics shift. Confirm against code.claude.com/docs/en/memory.
Stage 1 done when
  • Your repo has a committed CLAUDE.md with real commands, conventions, and architecture.
  • It contains nothing generic, no one-off task notes, and no style rules a linter should own.
  • It’s under ~200 lines — and you know what you’d split out if it grew.
Stage 02

Layer the configuration hierarchy

Claude Code Configuration & Workflows · 20%

One CLAUDE.md rarely fits a whole team and a whole codebase. Claude Code reads a hierarchy of memory files, from your personal preferences down to rules that fire only in specific folders. Understanding what wins when they conflict is exactly what the exam probes.

2a · The layers, broad to specific

Claude Code walks up the directory tree and loads every CLAUDE.md it finds, plus rules. In order of increasing specificity:

  • User — ~/.claude/CLAUDE.md: your personal preferences across all projects. Never committed. (“When no project style exists, I prefer explicit error handling.”)
  • Project — ./CLAUDE.md: team-wide, committed via git. The Stage 1 file.
  • Path-scoped rules — .claude/rules/*.md: load conditionally by paths: glob. Same priority as the project CLAUDE.md, but scoped.
  • Local — CLAUDE.local.md: your personal, project-specific notes; auto-gitignored.
  • Subdirectory — a CLAUDE.md deeper in the tree loads on demand when Claude reads files there.

Precedence: more specific wins. If your user file says “2-space indent” and the project file says “4-space,” the project instruction wins for that project.

2b · Path-scoped rules for area-specific instructions

When a rule only matters for one part of the codebase, scope it — so it loads only when Claude touches those files, keeping every other session’s context lean.

.claude/rules/api.md
--- paths: - "api/**/*.py" --- # API-layer rules - Every endpoint validates input before touching core logic. - Use the standard error envelope: {code, message, retryable}. - Never call `store/` directly — route through `core/`.
A rule with no paths: field loads unconditionally — the usual reason a rule that should be scoped is firing everywhere. And .claude/rules/*.md share priority with .claude/CLAUDE.md; they don’t override it.

2c · Diagnose a “why isn’t my rule applying?” problem

This is a core exam skill. When behavior diverges — a teammate isn’t getting an instruction, or a rule fires in the wrong place — check scope first:

  • Teammate not receiving an instruction? It’s probably in your user file or CLAUDE.local.md, which aren’t shared — move it to the project CLAUDE.md.
  • Rule firing everywhere? It’s missing a paths: field.
  • Instruction vanished after /compact? It was given in conversation only, or lives in a nested CLAUDE.md that hasn’t reloaded — put it in the project file to persist.
  • Use /memory to see exactly which files are loaded and in what order before concluding a rule is ignored.
Exam lens This domain is tested heavily as hierarchy diagnosis: a scenario describes conflicting or missing instructions and asks which level wins or where the fix goes. The keys: more-specific-wins, user is personal / project is shared / path-rules are scoped, and /memory is the debugging tool. Know when to reach for .claude/rules/ versus a directory-level CLAUDE.md.
Verify The layer set and precedence are current, but exact filenames (CLAUDE.local.md), the paths: frontmatter, and the managed/enterprise layers evolve. Confirm the full resolution order against code.claude.com/docs/en/memory.
Stage 2 done when
  • You have a user-level preference, a project CLAUDE.md, and at least one path-scoped rule that fires only in its folder.
  • You’ve deliberately created a conflict and confirmed the more-specific instruction wins.
  • You can use /memory to explain which files loaded and why.
Stage 03

Build the four team slash commands

Claude Code Configuration & Workflows · 20%

Your team does four things with Claude Code daily: generate, refactor, debug, document. Turn each into a slash command — a saved prompt anyone invokes with six keystrokes, committed to the repo so the whole team shares the same workflow.

3a · The shape of a command

A command is a markdown file whose body is the prompt and whose filename is the command name. Optional YAML frontmatter tunes it: description (shown in the picker), allowed-tools (restricts scope), argument-hint, and model (pin a model for the task). $ARGUMENTS holds whatever you type after the command.

Location note: .claude/commands/*.md is the original format and still works; the current recommended format is .claude/skills/<name>/SKILL.md, which supports the same /name invocation plus autonomous use by Claude. This lab shows the .claude/commands/ form for clarity — the frontmatter is the same either way.

3b · The four commands

Each maps a team task to a command, with the frontmatter doing real work — note how allowed-tools and model encode intent.

.claude/commands/generate.md
--- description: Generate a new module or function to spec argument-hint: [what to build] allowed-tools: Read, Edit, Bash(pytest *) --- Generate: $ARGUMENTS Follow the conventions in CLAUDE.md. Write the code, add tests next to it as test_*.py, and run pytest to confirm they pass.
.claude/commands/refactor.md
--- description: Refactor code without changing behavior argument-hint: [file or module] allowed-tools: Read, Edit, Bash(pytest *) model: opus # design work — pin the stronger model --- Refactor $ARGUMENTS for clarity and structure. Do NOT change behavior. Run the tests before and after to prove equivalence.
.claude/commands/debug.md
--- description: Reproduce and fix a bug argument-hint: [bug description or issue #] allowed-tools: Read, Edit, Bash --- Debug: $ARGUMENTS 1. Write a failing test that reproduces it. 2. Find the root cause — explain it before fixing. 3. Fix, then confirm the test passes and nothing else broke.
.claude/commands/document.md
--- description: Add or update documentation for code argument-hint: [file or module] allowed-tools: Read, Edit model: haiku # mechanical work — cheaper model is plenty --- Document $ARGUMENTS: docstrings on public functions, and a short module header explaining its role. Match the existing doc style.

3c · Why the frontmatter matters

  • allowed-tools declares intent. /document lists only Read, Edit — it can’t run shell commands, so it can’t accidentally do more than document. A narrow surface is a safer command.
  • model matches the task. Mechanical work (docs) runs fine on Haiku at a fraction of the cost; design work (refactor) pins Opus so nobody invokes it on a downgraded model and gets a worse result than it was tuned for.
  • Namespacing: subfolders group commands — .claude/commands/release/notes.md becomes /release:notes.
Exam lens Expect questions on frontmatter (what allowed-tools restricts, what $ARGUMENTS captures), on scope (project .claude/commands/ shared vs. ~/.claude/commands/ personal), and on the commands-are-now-skills convergence. Model-pinning per command (Haiku for mechanical, Opus for design) is a favorite “which is most cost-effective” item.
Verify The .claude/commands/ vs .claude/skills/ split, frontmatter keys, and whether model takes a friendly name or full ID all move between versions. Confirm against the slash-commands and skills docs before committing team-wide.
Stage 3 done when
  • All four commands exist, are committed, and run on your repo.
  • At least one restricts allowed-tools and one pins a model to fit the task.
  • You’ve invoked one with an argument and seen $ARGUMENTS substituted.
Stage 04

Plan mode vs. direct execution

Claude Code Configuration & Workflows · 20%

Knowing when to plan is one of the highest-value judgments in the whole tool. Plan mode makes Claude investigate and propose before touching anything; direct execution just does the work. Use the wrong one and you either add friction to a typo fix or let a risky refactor run unsupervised.

“Fix this type annotation” wants direct execution — one obvious change. “Restructure the API layer to add input validation everywhere” wants a plan you approve first, or Claude may reorganize things in a direction you didn’t want.

4a · What plan mode is

Plan mode is a read-only permission mode: Claude reads, searches, and proposes a step-by-step plan, but the read-only constraint is enforced at the tool level — it cannot edit files or run commands until you approve. Enter it three ways:

# 1. Mid-session: cycle permission modes Shift+Tab # default → acceptEdits → plan (watch the status bar) # 2. Slash command /plan # 3. Start the session in plan mode claude --permission-mode plan
On some terminals Shift+Tab only toggles two modes; use /plan or check your key bindings. The status bar shows ⏸ plan mode on when it’s active — watch it rather than assuming one keypress landed.

4b · The decision, as a rule

The heuristic worth making muscle memory: if you can already picture the diff, execute directly; if you can’t, plan first. Concretely:

  • Plan first when: the change touches many files, it’s a refactor crossing module boundaries, it’s a codebase you don’t know, it hits auth / billing / migrations, or the instruction is ambiguous. The plan is cheaper than the rework.
  • Execute directly when: it’s a single-file change, it’s trivially reversible (a git revert fixes it), you’ve done this shape of task before, or you’re in a tight iteration loop.

4c · The Explore → Plan → Execute pattern

For a real feature, run the full loop: let Claude explore (read the relevant code in plan mode), have it plan (a specific, file-by-file proposal you can edit — open it with Ctrl+G and tighten it), then approve and execute. Because it explored and you approved the approach, execution has no surprises. Plan mode also reads your CLAUDE.md during exploration, so your conventions shape the plan.

A subtlety worth internalizing: plan mode prevents accidental writes during planning — it does not stop you approving a bad plan. The judgment is still yours; the mode just makes the risky part visible before any file changes.
Exam lens Scenario-heavy: a task is described, you choose plan or direct. Map to the heuristic — wide/risky/unfamiliar/ambiguous → plan; small/reversible/known → execute. Know plan mode is a true tool-level read-only sandbox, the three ways to enter it, and that it reads CLAUDE.md. The tempting wrong answers either plan everything (friction) or execute everything (risk).
Verify The Shift+Tab cycle order, /plan, the --permission-mode plan flag, Ctrl+G to edit the plan, and model eligibility for plan mode are current but shift between versions and platforms. Confirm against code.claude.com/docs/en/permission-modes.
Stage 4 done when
  • You’ve run a trivial task with direct execution and a multi-file task through plan mode.
  • You reviewed and edited a proposed plan before approving it.
  • You can state the plan-vs-execute rule in one sentence and defend it on three examples.
Stage 05

Manage context on a long session

Context Management & Reliability · 15%

A long working session — hours of generating, refactoring, debugging — fills the context window. CLAUDE.md, every file read, the whole conversation all consume it, and as it fills, Claude’s adherence to your instructions drops. Keep the working context lean.

5a · See what you’re spending

You can’t manage what you can’t see. /context shows how much of the window is in use and where it’s going — CLAUDE.md, tool results, conversation. Check it when a long session starts feeling less sharp.

/context # show current context usage and breakdown /memory # show which instruction files are loaded, in order

5b · Compact, clear, and what survives

Two levers when the window fills:

  • /compact compresses the conversation to free space. Crucially, project-root CLAUDE.md survives — Claude re-reads it from disk and re-injects it. Instructions given only in conversation do not survive; if a rule matters, it belongs in CLAUDE.md.
  • /clear wipes the conversation entirely for a fresh start — use it between unrelated tasks so old context doesn’t bleed in.
This is the answer to a classic confusion: “my instruction disappeared after /compact.” It was conversation-only, or in a nested CLAUDE.md that hasn’t reloaded. The fix is to write persistent rules into the project CLAUDE.md, not to re-type them each session.

5c · Delegate investigation to a subagent

When a task needs Claude to read widely — “how is error handling done across this codebase?” — that exploration can flood your main context with twenty files. Let Claude spawn an Explore subagent: it reads in its own context and returns a concise summary, so the main session gets the finding, not the raw files. Check /context after — usage stays minimal despite a thorough investigation.

5d · A scratchpad for long work

For multi-hour work, keep durable state on disk — a short notes or task file Claude updates — rather than trusting it all to stay in the window. State lives in the file; the window holds only what the current step needs. It also means a /clear or a new session can pick up where you left off.

Exam lens Scenarios describe a session degrading over time; you pick the fix. The cluster: observe with /context, compact/clear to reclaim space, delegate exploration to a subagent for isolation, and external state over in-context memory. Know what survives compaction (project CLAUDE.md) and what doesn’t (conversation-only instructions) — that exact distinction shows up.
Verify /context, /compact, /clear, /memory, and the compaction-survival behavior are current but evolving (auto-compaction, auto memory). Confirm against code.claude.com/docs/en/memory and the CLI reference.
Stage 5 done when
  • You’ve read /context on a real session and can say where the tokens went.
  • You’ve compacted and confirmed your CLAUDE.md rules stayed active afterward.
  • You delegated a wide investigation to a subagent and saw the main context stay small.
Stage 06

Add guardrails that always hold

Context Management & Reliability · 15%

CLAUDE.md is guidance — Claude usually follows it, but it’s context, not a wall. For things that must never happen (no edits to generated files, no committing secrets, tests must pass before a commit), you need enforcement that holds regardless of what Claude decides. That’s hooks and settings.

6a · Guidance vs. enforcement

The distinction the exam leans on:

  • CLAUDE.md = guidance. Loaded as context; strongly influences behavior; can be crowded out or not followed perfectly. Right for conventions and preferences.
  • Hooks & settings.json rules = enforcement. Evaluated by the client regardless of the model’s reasoning. A hook’s deny holds even in permissive modes. Right for guarantees.

If a scenario says “ensure X never happens,” the answer is a hook or a deny rule — not a line in CLAUDE.md.

6b · A PreToolUse hook that blocks the unwanted

A PreToolUse hook runs before any tool call and can deny it. Use one to protect generated files, lockfiles, or credential paths — a guarantee that holds no matter what the model tries.

# .claude/settings.json — block edits to generated files { "hooks": { "PreToolUse": [{ "matcher": "Edit|Write", "hooks": [{ "type": "command", "command": "block-if-path-matches '**/*.generated.*' '**/lock'" }] }] } }

6c · A quality gate with a Stop hook

Hooks also fire at other lifecycle points. A Stop hook can run your linter and formatter automatically when Claude finishes, so mechanical quality is enforced deterministically rather than hoped for in CLAUDE.md — which the docs explicitly recommend over asking Claude to lint.

# run lint + format every time Claude wraps up { "hooks": { "Stop": [{ "hooks": [{ "type": "command", "command": "ruff check --fix . && ruff format ." }] }] } }
Exam lens The single most tested idea in this pair of domains: guidance vs. enforcement. CLAUDE.md influences; hooks and deny rules guarantee. “Put a hard constraint in CLAUDE.md” is the classic wrong answer. Also know: use deterministic tools (linters via hooks) for mechanical rules rather than spending CLAUDE.md’s instruction budget on them.
Verify Hook event names (PreToolUse, Stop), the settings.json hook shape, and matcher syntax are current but version-sensitive. The block-if-path-matches command above is illustrative — write a real script or use a supported matcher. Confirm against code.claude.com/docs/en/hooks.
Stage 6 done when
  • You can state which belongs in CLAUDE.md and which needs a hook, with an example of each.
  • A PreToolUse hook denies an edit you want blocked — and holds even in acceptEdits mode.
  • A lint/format step runs automatically rather than living as a CLAUDE.md instruction.
Stage 07

Ship it to the team

Workflow in practice

The whole point of this setup is that the team inherits it. A workflow that lives only on your machine helps one person; committed to the repo, it standardizes how everyone works. Commit the shared parts, keep the personal parts personal, and confirm a teammate gets what you intended.

7a · Commit the shared config

These belong in version control so every teammate picks them up on clone:

  • ./CLAUDE.md — the project context.
  • .claude/rules/*.md — path-scoped rules.
  • .claude/commands/*.md (or .claude/skills/) — the four commands.
  • .claude/settings.json — the hooks and guardrails.

7b · Keep personal things out of the shared config

These are per-developer and must not be committed: your ~/.claude/CLAUDE.md preferences (they live in your home dir, not the repo) and any CLAUDE.local.md (auto-gitignored). If a rule needs to reach the whole team, it goes in the project file — the Stage 2 diagnosis in reverse.

7c · Verify a teammate gets it

Confirm the setup actually transfers: have a teammate (or a fresh clone) run claude in the repo, then /memory to see the project CLAUDE.md and rules load, and /help to see the four commands appear. If something’s missing, it’s almost always a scope problem — it was in a personal file, not the committed one.

Exam lens The recurring theme: shared vs. personal scope. Project files (committed) reach the team; user and local files don’t. A “teammate isn’t getting the behavior” scenario is nearly always a rule sitting in a personal file that should be in the project config. Know which files are committed and which are gitignored.
Stage 7 done when
  • CLAUDE.md, rules, commands, and settings are committed; personal preferences are not.
  • A fresh clone loads the project context and shows the four commands.
  • You can explain, for any instruction, whether it should be shared or personal — and where it goes.
Wrap-up

What you built, and how it maps

You now have one real repo with a complete Claude Code workflow on it, exercising the scenario’s two primary domains: Claude Code Configuration & Workflows (20%) — a layered CLAUDE.md hierarchy, path-scoped rules, four team slash commands with intent-encoding frontmatter, and the plan-vs-execute judgment — and Context Management & Reliability (15%) — observing and reclaiming context, delegating investigation, and enforcing guarantees with hooks rather than hoping CLAUDE.md holds. Completing it, you should be able to answer any question the scenario raises: what wins when configs conflict, when to plan versus execute, what survives compaction, and where a hard rule belongs.

The exam is 60 scenario questions in 120 minutes, and it rewards judgment over recall — the wrong answers are written to tempt people with partial knowledge. Having actually configured the hierarchy, felt the plan-vs-execute tradeoff, and enforced a guardrail with a hook is the difference between recognizing the right call and guessing.
